### Findings — issues to address

#### S-1: CSP allows `'unsafe-inline'` on `script-src` and `style-src` (severity: medium)

**Location:** `next.config.ts:18-19`

The `Content-Security-Policy` response header sets `script-src 'self' 'unsafe-inline'` and `style-src 'self' 'unsafe-inline'`. `'unsafe-inline'` defeats CSP's main XSS mitigation: it allows any inline `<script>` or event handler to execute, so a future XSS sink would not be contained by the policy. Lighthouse `csp-xss` flags this **High** severity (see [Lighthouse table](#lighthouse--non-performance-categories) above).

Practical impact today is low — there are no XSS sinks, no third-party script origins, and no `eval`. The risk is purely defense-in-depth.

**Remediation:** Adopt nonce-based CSP via Next.js middleware ([guide](https://nextjs.org/docs/app/guides/content-security-policy)). The inline-comment at `next.config.ts:14-15` already flags this as deferred; lift it. Same approach also lets `'unsafe-inline'` come off `style-src` (use a hash or move all inline `<style>` to the static stylesheet).

#### S-2: Hardening headers missing (severity: low)

**Location:** `next.config.ts:4-26`

Set: HSTS (preload), `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`, `frame-ancestors 'none'`.

Missing:
- `Permissions-Policy` — currently the browser grants the page default access to camera/mic/geolocation/etc., even though the app uses none of them. A compromised script could try to call them.
- `Cross-Origin-Opener-Policy: same-origin` — defends against cross-origin window leaks (Spectre-class).
- `Cross-Origin-Resource-Policy: same-origin` — prevents the page being loaded as a subresource by other origins.

**Remediation:** Add to the `securityHeaders` array. Suggested values:

```ts
{ key: "Permissions-Policy", value: "camera=(), microphone=(), geolocation=(), payment=(), usb=()" },
{ key: "Cross-Origin-Opener-Policy", value: "same-origin" },
{ key: "Cross-Origin-Resource-Policy", value: "same-origin" },
```

### Findings — forward-compat hardening (currently zero impact)

#### S-3: No CSRF protection on mutating endpoints

**Location:** `app/api/todos/route.ts` (POST), `app/api/todos/[id]/route.ts` (PATCH, DELETE)

Currently no auth, no cookies, no per-user state — so CSRF moves no attacker value. Two layers of incidental protection exist:
- `req.json()` requires JSON body. Simple `<form>`-based CSRF (`application/x-www-form-urlencoded`) fails Zod parsing.
- The `userId` filter is `null` for everyone, so any cross-origin caller is acting on the same shared dataset they could already reach directly.

**Once auth ships**, this becomes a real CSRF surface. Mitigation should be in place at the moment cookies are introduced: `SameSite=Lax` (or `Strict`) on session cookies + Origin/Referer check on POST/PATCH/DELETE, or a CSRF token middleware. Document this as a **blocker for the auth story**.

#### S-4: No rate limiting or abuse protection

**Location:** All `app/api/todos/*` routes

Anonymous unlimited writes — an attacker can fill the database with 280-char rows. Per-row size is bounded (~300 bytes including UUID and timestamp); 1M rows ≈ 300 MB. Acceptable for v1 because the deployment is single-tenant and protected by Neon's branch-level quotas, but it's free abuse if the URL leaks.

**Remediation when going public:** Per-IP rate limit on POST `/api/todos`. Options: Vercel BotID, Upstash Redis + a small middleware, or Vercel's built-in rate-limit middleware once enabled on the project.

#### S-5: Sentry tracesSampleRate at 1.0

**Location:** `sentry.server.config.ts:10`, `sentry.client.config.ts:10`

Not a security issue — flagging for completeness because high-volume tracing in production can become a billing surprise. `sendDefaultPii` is not set (defaults to false), so PII handling is correct.

### Validation contract reference

For future work, the request-side trust boundary is enforced exclusively by:

| Schema | Location | Constraint |
|---|---|---|
| `TodoCreateSchema` | `lib/validation.ts:3-6` | `id: uuid`, `description: trimmed string 1..280` |
| `TodoUpdateSchema` | `lib/validation.ts:9-11` | `completed: boolean` |
| `IdSchema` (route param) | `app/api/todos/[id]/route.ts:6` | `uuid` |

Any new API route must follow the same pattern — `safeParse` first, return `validationFailed(parsed.error.message)` on failure. The Zod message is returned verbatim; safe today (only the values the client just sent) but worth re-evaluating if schemas later include server-side fields.
