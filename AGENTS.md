# Project conventions

This project follows the architecture and UX specifications under `_bmad-output/planning-artifacts/`. When in doubt, those documents win — this file is a fast-reference summary.

## Naming

- **React components:** PascalCase identifier and PascalCase filename (e.g., `TaskItem.tsx` exports `TaskItem`).
- **shadcn primitives:** kebab-case filenames in `components/ui/` (e.g., `button.tsx`). Generated by the shadcn CLI — **do not rename**.
- **Hooks:** kebab-case filename, camelCase identifier prefixed with `use` (e.g., `hooks/use-create-todo.ts` exports `useCreateTodo`).
- **Non-component modules:** kebab-case (e.g., `lib/api-client.ts`, `db/queries.ts`).
- **Functions and variables:** `camelCase`. **Constants:** `SCREAMING_SNAKE_CASE` for exported configuration (e.g., `UNDO_TIMEOUT_MS`).
- **Types and Zod schemas:** PascalCase; Zod schemas end in `Schema` (e.g., `TodoCreateSchema`); inferred type drops the suffix (`type TodoCreateInput = z.infer<typeof TodoCreateSchema>`).

## Import graph (enforced by ESLint `no-restricted-imports`)

```
   app/api/* ──► db/queries.ts ──► db/client.ts
                                        │
   app/page.tsx ──► components/* ──► hooks/* ──► lib/api-client.ts
                          │                              │
                          └──── lib/validation.ts ◄──────┘
```

- `components/**` MUST NOT import from `db/**` or `app/api/**`.
- `hooks/**` MUST NOT import from `db/**` or `components/**`.
- `app/api/**` MUST NOT import from `components/**` or `hooks/**`.

If lint fails on `no-restricted-imports`, the import graph is wrong — do not silence the rule, restructure the code.

## No modals, ever (UX policy)

- `Dialog`, `AlertDialog`, `Modal`, `confirm()`, `alert()`, `prompt()` — all forbidden.
- Destructive actions use the `UndoToast` deferred-action pattern (Story 3.2).
- Errors surface inline as `ErrorIndicator` (Story 4.1).
- ESLint blocks imports from `@radix-ui/react-dialog`, `@radix-ui/react-alert-dialog`, `@base-ui/react/dialog`, `@base-ui/react/alert-dialog`.

## Design tokens

All color and spacing values live as CSS variables in `app/globals.css`. **Never hardcode hex values or pixel sizes in components.** Use Tailwind utilities that resolve to the CSS variables:

| Token (CSS var) | Tailwind utility |
| --- | --- |
| `--bg` | `bg-background` |
| `--surface` | `bg-surface` |
| `--foreground` | `text-foreground` |
| `--foreground-muted` | `text-foreground-muted` |
| `--border-subtle` | `border-border-subtle` |
| `--accent` | `bg-accent` / `text-accent` (also wired to `--primary` for shadcn primitives) |
| `--accent-foreground` | `text-accent-foreground` |
| `--error-foreground` | `text-error-foreground` |
| `--space-1` … `--space-8` | Tailwind's default 4px-grid spacing (`p-1`, `gap-3`, `mt-6`, etc.) |

The shadcn primitives in `components/ui/*` reference shadcn's internal token names (`--primary`, `--ring`, `--muted-foreground`, etc.); those names are aliased onto our v1 tokens in `globals.css`, so visual changes happen by editing the v1 tokens — not the alias mappings.

## Database

- **Schema source of truth:** `db/schema.ts`. Every column type, constraint, and default lives there. Migrations are **generated** with `pnpm db:generate`, never hand-written. Drift between `schema.ts` and the produced SQL is a bug — regenerate, don't patch.
- **Driver:** `pg` (node-postgres) via `drizzle-orm/node-postgres`. **Never** `drizzle-orm/neon-http` — the architecture chose the Node driver for richer query support and easier local Postgres development.
- **`db/client.ts` is `server-only`.** First line is `import "server-only";`. Never import `db/client` (or anything that re-exports it) from `components/`, `hooks/`, or `lib/api-client.ts`.
- **All queries are parameterized Drizzle expressions.** Raw SQL strings in application code are forbidden. The only acceptable `sql\`...\`` template is for DDL in tests (e.g., `TRUNCATE TABLE todos`) where Drizzle's expression API has no helper.
- **`userId` parameter is mandatory** on every query helper, even though v1 always passes `null`. Forward-compat for auth — do not "simplify" by removing it.
- **Migrations run before `next build`.** The unified `pnpm build` script is `drizzle-kit migrate && next build`. A migration failure aborts the deploy before serving traffic. On Vercel, `DATABASE_URL` (and `DATABASE_URL_UNPOOLED`, used by `drizzle.config.ts` for migrations when present) come from the Vercel-Neon Marketplace integration. Locally, pull them via `pnpm dlx vercel env pull .env.local`.
- **Test-DB strategy:** integration tests in `*.test.ts(x)` run against the same `DATABASE_URL` set in `.env.local` — typically a personal Neon dev branch. `beforeEach` truncates the affected tables. No mocking layer for DB tests; the real schema is the contract.

## Migrations

- `pnpm db:generate` — produce a migration after editing `db/schema.ts`. Drizzle-kit names files `0000_<adjective>_<noun>.sql`; the `0000_` ordering prefix is what matters, the suffix is auto-generated.
- `pnpm db:migrate` — apply all pending migrations against `DATABASE_URL_UNPOOLED` (or `DATABASE_URL` if unpooled is unset). Idempotent.
- Both commands read `DATABASE_URL_UNPOOLED` first, falling back to `DATABASE_URL`. For Neon, the pooled endpoint can be unfriendly to migration DDL in transaction mode; the unpooled (direct) endpoint is the safer default.
- Generated SQL files in `db/migrations/` are committed and **never hand-edited**. To change a migration, edit `db/schema.ts` and regenerate.

## Tooling

- **Package manager:** `pnpm`. Never `npm` or `yarn`.
- **Node version:** 22 LTS (see `.nvmrc`). Local development on a different version is allowed but Vercel uses 22; reproduce CI failures locally on 22.
- **Type-checking:** `pnpm typecheck` (`tsc --noEmit`). Required green before any PR.
- **Linting:** `pnpm lint`. Required green before any PR. Story 4.3 will tighten `jsx-a11y` warnings into CI-failing errors.
- **Tests:** Vitest for unit/integration (Story 1.2 onward), Playwright for E2E (Story 1.5+).

## Deviations from the architecture document

The architecture document (`_bmad-output/planning-artifacts/architecture.md`) was authored against Tailwind 3 and assumes a `tailwind.config.ts` file. This project ships **Tailwind 4** (the Next.js 16 default), which uses a CSS-first configuration via the `@theme` directive in `app/globals.css`. **There is no `tailwind.config.ts`** — design-token mapping lives entirely in `globals.css`.

If a future story requires `tailwind.config.ts` (for plugins, content scanning, etc.), introduce it then; otherwise the CSS-first approach is the project standard.
